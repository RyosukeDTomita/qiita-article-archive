## 生成AI時代に重要な力は好奇心である

生成AIの登場によって「もうコードを書けるだけでは価値が出ない」と言われるようになった。
では、次に何が重要になるのだろうか。

よく挙げられるのが「好奇心」である[^1][^2]。

生成AIはコードを書くことはできるが、「何を作るか」「どこを自動化するか」「その結果に責任を持つか」といった判断は、現時点では人間が行う必要がある。

だからこそ重要になるのが、

- このやり方、もっと楽にできないのでは？
- そもそもこの作業、自動化できるのでは？

といった問いを立てる力であり、その源泉として好奇心がある。

### 好奇心は可視化できない。手を動かした記録である「継続力」で差をつけろ!

しかし、好奇心は可視化できない。

生成AI時代には企画を考えるだけでなく、動くものが求められるケース[^3]が今後増えると予想される。

大きくバズったOSSや派手な実績がない新人・ジュニアエンジニアは
「問いを立て、実装までやり切る」ことを継続していることを示すことができれば、それは十分に好奇心の証拠になるのではないだろうか。

## GitHubの草を毎日生やすことを目標にしてみたら毎日学習が継続できた

:::note warn
※n=1の話なので注意
:::

上の前置きのようなことを考えたので、ここから実践してみた話を書く。

なんでもいいのでGitHubにpushするということを始めた(この記事執筆時点で126日連続でGitHubに草を生やせている)。

![image.jpeg](https://qiita-image-store.s3.ap-northeast-1.amazonaws.com/0/3718390/390981ac-57ff-4174-9ce1-31dd259dd2f0.jpeg)

とりあえず、続けることが大切なので厳密に[Write Code Everyday](https://johnresig.com/blog/write-code-every-day/)をするのではなく、

- privateリポジトリへの投稿でもOK
- ちょっとしたメモだけでもOK

という制約ゆるゆるで自分はやっている。

### やるき継続のために、ストリークを計算してREADMEに表示するバッチを導入したが、不安定だった。

ルールがゆるゆるでもサボりたくなる日はあるので、連続して何日連続でコミットできたか(ストリーク)を計算することにした。
これにより、過去積み上げたストリークがサンクコストになり、とりあえず毎日継続できるようになった。

そうなると、何日継続しているかを定期的に確認したくなるので、[github-readme-streak-stats](https://github.com/DenverCoder1/github-readme-streak-stats)を使って作成したバッチをGitHubのProfile READMEに貼り付けてみた↓イメージ

![image.png](https://qiita-image-store.s3.ap-northeast-1.amazonaws.com/0/3718390/e9cef968-b505-464d-bf1e-bcbef48bbb5a.png)

UIもかっこいいし、無料で使わせていただけるのだが、

- URLのクエリパラメータでユーザを指定する
- 開発者の方がホストしてくださっているサイト経由でSVGを都度生成している

という関係上、svgが表示されなくなってしまうことが多々あり、結局GitHubの草を見に行くことが多くなってきた。

### ストリークだけを見る、最低限の機能で自作してみた

せっかくストリークがつながっていても表示できなければ、自分が確認してニヤニヤするためにも、他人にアピールするためにも使えなくて不便なので自分でバッチを作ってみた。

https://github.com/RyosukeDTomita/github-grass-streak

↓イメージ
![image.png](https://qiita-image-store.s3.ap-northeast-1.amazonaws.com/0/3718390/ce73195b-a906-4776-8740-a4988d5fed84.png)

安定して表示できるようにすることが最重要だったので以下のような構成にした。

1. GitHub Actionsで毎日9時にスクリプトを実行し、バッチを生成する
2. 生成したバッチをGitHub Pagesでホストする

バッチの更新は1日1回だけだが、安定して表示されるようになった。

同じような悩みの方がもしいれば使ってみてください!!!

## おまけ: 自分のコミット内容

先程もゆるゆると書いたが、以下のような内容をコミットしている。
難易度(個人的なハードルのでかさ)をつけてみた。

- 競技プログラミングの問題を解いた解答: 難易度中
- Obsidianに読んだ技術記事や書籍の内容をまとめる(private repository): 難易度低
- 読んでいる技術書のサンプルコードがGitHubにあるならフォークして自分用にコメントをつける: 難易度低
- 新しいOSSを作る: 難易度高
- 既存のOSSにコミット: Very Hard
- 新しいフレームワークとか言語を試して環境構築手順をまとめる: 難易度中〜高

開発自体はもともと好だったが、毎日継続する仕組みを作ることで、前よりもインプットもアウトプットも増やせたと思う。

去年は継続が目標になっているところがあったので、今年は新しいOSSを作ったり、使っているOSSにコミットを増やしていきたい。

今年も頑張るぞ!
